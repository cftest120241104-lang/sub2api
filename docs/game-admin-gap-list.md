# 游戏后台缺口清单

本文档按“可上线游戏后台”口径列出当前后台还需要补齐的能力。后台目标不是订阅/API 密钥平台，而是支持多渠道、多游戏、多场景调试和运营结算。

## 已具备

1. 游戏运营主导航
   - 已有仪表盘、渠道管理、游戏管理、渠道游戏配置、玩家/会话、Spin/注单、游戏服务调试入口。
   - 可见页面已从通用上游管理台收敛为游戏运营语义。

2. 游戏服务联调
   - 可访问 `pragmaticplay_games/server` 的健康检查、就绪检查、会话创建、GM 场景、Spin/Collect 调试。
   - 协议层已约定服务端业务字段使用 `nextStep`、`winCents`、`totalWinCents`、`symbols`，Pragmatic 短字段只保留在兼容边界。

3. 多渠道多游戏基础视图
   - 已有渠道、游戏、渠道游戏配置、玩家汇总和注单汇总页面。
   - 当前部分配置仍以前端本地 mock/localStorage 为主，适合作为交互样板，不满足生产持久化。

## P0 上线前必须补齐

1. 后台配置持久化
   - 渠道、游戏、渠道游戏配置必须落 PostgreSQL。
   - 禁止生产依赖 localStorage/mock 数据。
   - 配置变更必须写审计日志。

2. 权限与租户隔离
   - 至少区分平台管理员、运营、渠道管理员、只读审计角色。
   - 渠道管理员只能看本渠道玩家、注单、报表和配置。
   - 所有后台 API 必须服务端鉴权，不能只靠前端隐藏菜单。

3. 场景配置中心
   - 服务端需要一份可配置场景表：不中奖、普通中奖、大奖、Jackpot、FreeSpin、Bonus、异常/退款场景。
   - 每个场景需定义 `gameCode`、`scenarioId`、赔率/派彩、`nextStep`、屏幕符号、是否需要 collect。
   - GM 只能选择已启用场景，操作写审计。

4. 协议与路由边界
   - 客户端优先使用项目 JSON 路由：`/api/games/:gameCode/state`、`/spins`、`/collect`。
   - Pragmatic KV 路由 `/gs2c/ge/v3/gameService` 只作为外部兼容入口。
   - 短字段 `na/w/tw/s/sa/sb` 只允许出现在协议适配层和回归测试里。

5. 运营报表
   - 按时间、渠道、游戏统计总下注、总派彩、GGR、RTP、局数、活跃玩家。
   - 支持 CSV 导出，并保证导出数据与列表筛选一致。

## P1 二开增强

1. 游戏分支和发布管理
   - 每个 `gameCode` 支持多版本分支。
   - 渠道可绑定指定版本，支持灰度和回滚。

2. 风控策略
   - 支持玩家/渠道限额、异常连赢、异常下注频率、余额异常等规则。
   - 命中策略进入风控事件列表。

3. 钱包与结算
   - 钱包流水支持 DEBIT、CREDIT、REFUND 全链路追踪。
   - 渠道结算按分成比例生成报表。

4. 告警与健康监控
   - 对服务不可用、数据库异常、Redis 异常、Spin 错误率升高、RTP 偏离设置告警。
   - 支持 webhook/邮件扩展。

## P2 可后置

1. 活动与任务系统
   - 活动时间窗、预算、奖励发放和效果统计。

2. 操作审批
   - 高风险配置变更增加审批流。

3. 上游通用底座物理清理
   - 删除订阅、API 密钥、AI 账号、支付等不属于游戏运营后台的后端模块和前端组件。
   - 该项影响 Go ent/wire/generated 与大量前端类型，需单独分支迁移。

## 验收口径

- Docker Desktop 中容器按项目归类，后台容器名为 `pragmaticplay-admin-web`，服务端为 `pragmaticplay-server`。
- 管理员登录后首屏和左侧导航不出现订阅、API 密钥、AI 网关等无关入口。
- 创建会话、选择 GM 场景、Spin、Collect 能在客户端和后台注单里闭环看到。
- 一次中奖场景返回既包含 Pragmatic 兼容短字段，也包含语义化字段；客户端使用语义化字段解析成功。
